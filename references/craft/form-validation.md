# 表单验证 — 状态机与时机规则

> 表单是用户完成任务的关键路径。验证体验直接影响转化率。

## 输入框状态机

```
pristine → dirty → touched → invalid-after-touched
                               ↓
                         valid-after-touched
                               ↓
              submitting（禁用所有输入）
                    ↓           ↓
              submitted    submit-error
```

## 状态定义

| 状态 | 触发条件 | 视觉表现 |
|------|----------|----------|
| pristine | 初始，未交互 | 默认样式 |
| dirty | 开始输入内容 | 无变化 |
| touched | 输入框失焦 | 开始验证 |
| invalid | 失焦后验证不通过 | 红色边框 + 红色错误信息 + 错误图标 |
| valid | 失焦后验证通过 | 绿色勾号（可选） |
| disabled | 提交中/无权限 | 灰色背景 + opacity 0.5 + cursor: not-allowed |
| submitting | 表单正在提交 | 提交按钮 spinner + 所有字段 disabled |
| submitted | 提交成功 | Toast 通知 2-3 秒自动消失 |
| submit-error | 服务端返回错误 | 表单顶部错误信息 + 字段级错误 |

## 验证时机规则（最重要的规则）

```
1. 不要在用户还在输入时实时报错
   → 打断心流，造成焦虑
   → 例外：密码强度指示器（正面反馈，非错误）

2. 首次验证在 blur（失焦）时触发
   → 用户离开字段时才告诉他有没有问题

3. blur 验证通过后，切换为实时验证
   → 用户回头修改已通过字段时，立即反馈变化
   → 因为用户已经知道规则，实时反馈是有帮助的

4. 提交时验证所有字段
   → 滚动到第一个错误字段
   → 错误字段获得焦点

5. 提交按钮在 submitting 状态禁用
   → 防止重复提交
   → 显示 spinner + "提交中..."
```

## 错误信息写法

```
❌ "格式错误"
✅ "邮箱格式不正确，请输入如 name@example.com"

❌ "此字段为必填"
✅ "请输入手机号码"

❌ "密码不符合要求"
✅ "密码需要至少 8 位，包含 1 个大写字母和 1 个数字"

规则：
- 说清楚"错在哪"+"怎么修"
- 用正面语言（"请输入"而非"不能为空"）
- 给出具体的格式示例
```

## 表单布局规则

```
标签位置：标签在输入框上方（不是左侧，不是内部）
  → 垂直视觉流，移动端友好
  → placeholder 用示例值（如 "name@example.com"），不是说明文字

字段间距：16px（相关字段）/ 24px（字段组之间）

长表单：拆分为多步 + 进度条
  → 每步不超过 5 个字段
  → 显示"第 2 步 / 共 4 步"
  → 利用 Zeigarnik Effect（未完成感驱动继续）

提交按钮：
  → 放在表单底部，左对齐（跟随阅读流）
  → 主操作在右/下方，取消在左/上方
  → 宽度可以是 100%（移动端）或 auto（桌面端）
```

## 常见验证规则

| 字段类型 | 验证规则 | 错误信息示例 |
|----------|----------|-------------|
| 邮箱 | 基本格式检查 | "请输入有效的邮箱地址，如 name@example.com" |
| 手机号 | 11 位数字 | "手机号为 11 位数字" |
| 密码 | 8+ 位，含大写+数字 | "密码至少 8 位，需包含大写字母和数字" |
| 确认密码 | 与密码一致 | "两次输入的密码不一致" |
| 必填 | 非空 | "请输入 XX" |
| 数字范围 | min/max | "金额范围为 1-100,000" |
| 文件上传 | 类型+大小 | "支持 JPG/PNG 格式，文件不超过 5MB" |
