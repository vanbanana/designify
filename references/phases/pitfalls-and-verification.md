# Pitfalls（必须避免的错误）

- 不要用户说"做网站"就直接写代码——必须先完成 Phase 1 提问
- 不要跳过 Design Read——必须在写代码前输出设计推断（Phase 3 → `output/design-read.md`）
- 不要只做一个"好看的首页"——必须按项目类型生成完整页面
- 不要用 lorem ipsum——必须用符合业务的真实中文内容
- 不要忽略五态覆盖——每个数据展示区域都要有 Loading/Empty/Error/Populated/Edge
- 不要用硬编码颜色——必须用 CSS 变量 + 60-30-10 比例
- 不要用 linear 缓动——所有动效必须用缓动曲线
- 不要用 div soup——必须用语义化 HTML 标签
- 不要用 emoji 当图标——必须用专业图标库（Lucide/Heroicons）
- **不要从零造轮子——图标/插画/图片必须从现成素材库引入，不要程序化生成 SVG**
- **不要用 5 个固定标签让用户选风格——让用户自由描述，AI 在内部映射到 8 个类别**
- 中文内容不要用小于 14px 的字号
- 不要所有模块都用同样的卡片样式——要有视觉节奏变化
- 不要只在对话里输出文档——必须写入 `output/` 目录下的对应文件
- 不要生成代码后就结束——必须执行 Phase 6 质量门禁 + Critique Theater

# Verification（最终确认）

Skill 执行完成后确认：
1. 是否完成了 Phase 1 的需求发现（而不是直接开写）？
2. 是否输出了 Design Read（Phase 3 的设计推断）？
3. 页面清单是否匹配项目类型标准？
4. 量化检查清单是否全部通过？
5. Anti-AI-Slop P0 检查是否全部通过？
6. 五态覆盖是否完整（每页 5/5）？
7. 三个响应式断点是否都检查过？
8. Critique Theater 是否执行了至少 1 轮自审？
9. 设计决策是否有心理学依据（能说出用了哪条定律）？
10. 用户是否清楚接下来该做什么？
