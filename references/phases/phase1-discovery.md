# Phase 1: 需求发现（提问协议）— 详细指令

**目标：通过 3 轮提问（含技术栈强制确定），帮用户从"我想做个XX"变成清晰的产品需求。**

---

## 第 1 轮：理解意图（必问）

用户说出需求后，先不要写任何代码。用以下问题框架理解用户真实意图：

```
必问（合并为 1 次回复，不超过 4 个问题）：

Q1. 你想做的是什么类型的产品？
   → 选项引导：网站/App/小程序/后台管理系统/桌面应用/AI产品/其他
   → 如果用户说不清楚，给出 3 个最接近的例子让他选

Q2. 这个产品主要给谁用？解决什么问题？
   → 用大白话解释："就是你的用户是谁，他们用这个干什么"
   → 如果用户答不上来，帮他推理："比如做电商的话，买家用来浏览商品下单，
     卖家用来管理商品和订单"

Q3. 你有没有觉得哪个网站/App的界面特别好看，或者想参考的？
   → 给几个跨风格的具体例子帮用户打开思路：
     "比如 Notion 那种干净简洁的、Stripe 那种专业商务的、
      Linear 那种暗色科技感的、Airbnb 那种温暖有质感的..."
   → 如果用户说出具体产品/URL → 记录，后续用 website-analyzer 工具提取设计系统
   → 如果用户说"没有/不知道" → 不追问，AI 主动调研同类产品（见下方"主动调研"）
   → 如果用户只给了模糊关键词（如"高级感"），用下方映射表推断

Q4. （**强制**）你希望用什么技术来开发？
   → 选择题引导：React/Next.js / Vue/Nuxt / 单文件 HTML / Tailwind CSS / 其他
   → 如果用户说"随便"或"你定"：
     - 是前端开发人员？→ React + Vite
     - 是小白用户？→ React + Tailwind（非纯 HTML！）
     - 是快速原型？→ 单文件 HTML
   → 记录到技术栈声明（Phase 2 产品定义文档的必填字段）
   → **铁律 7**：没有技术栈声明就不能写一行代码。Phase 1 必须确定框架/样式/构建方案。
```

## 参考网站分析

**触发条件（满足任一即执行）：**
- 用户在 Q3 中提到了具体产品名或 URL（如"像 Stripe 那样"、"linear.app"）
- 用户发来了截图并要求"参考这个风格"

**工具：** `tools/website-analyzer.mjs`（零外部依赖，纯 Node.js 内置 API，v3）
**用途：** 逆向提取任意网站的完整设计系统。核心技术：
- **反向 CSS 查找**（v3）：从 HTML 中提取实际 class 名（如 `item-wrap×138`），反查 CSS 精确规则。数据驱动，非随机选代表。实测 9/10 命中率。
- **复刻蓝图（Section 0）**：`:root` CSS 变量 + 页面骨架 + 色彩角色 + **15+ 种组件精确 CSS**（动态计数，因网站而异） + 7 步指南
- **HTML 结构分析（Section 2）**：导航项（从 `<a>` 文本提取，兼容 Vue/React SPA）、标题层级、卡片 class 模式、媒体统计
- **像素线框图（Section 11）**：ASCII 布局图 + 尺寸标注
- **截图（可选）**：`--screenshot` 参数，Jina AI + Chrome headless 双 fallback
- **CSS 令牌（Section 3-9）**：8000+ 条规则、颜色/排版/间距/断点/动效

**调用方式：**
```bash
node tools/website-analyzer.mjs <url> --output output/website-analysis.md --screenshot output/screenshot.png --verbose
```
**产物位置：** `output/website-analysis.md`（+ 可选 `output/screenshot.png`）
**下游消费：** Phase 3 Design Read 读取此报告。复刻时优先用 Section 0 的 `:root` 变量和 `class-*` 配方（来自反向查找的原站真实 CSS）。

### 分析结果验证（必须执行）

运行 analyzer 后，必须验证分析结果是否有效：

| 条件 | 判定 | 操作 |
|------|------|------|
| CSS 源 < 5KB 或设计令牌 Section 为空 | ❌ 分析失效 | 执行 Fallback |
| CSS 源 ≥ 5KB 且有设计令牌数据 | ✅ 有效 | 正常使用 |

**Fallback 流程（分析失效时）：**
1. 读取 `references/craft/platform-philosophy.md` 确定平台基线（C/B/G/IoT）
2. 根据 Phase 1 Q3 风格映射查找对应的品牌 DESIGN.md
3. 使用品牌 DESIGN.md 的设计令牌替代 analyzer 提取值
4. 在 Design Read 中注明："website-analyzer 分析结果无效，使用 [品牌] 设计系统作为替代"

```bash
# 快速验证命令
ls -la output/website-analysis.md
# 如果文件 < 5KB 或不存在，执行 Fallback
```

## 主动调研（无参考 URL 时执行）

**触发条件：** 用户没有提供任何参考网站或产品名。

**执行步骤：**
1. 根据产品类型，在 Mobbin / Refero / SaaSFrame 等灵感站搜索同类产品截图
2. 选取 2-3 个公认设计优秀的同类产品作为隐性参考
3. 加载对应的品牌 DESIGN.md（从 74 个品牌库中匹配）
4. 综合得出设计方向，在 Design Read 中说明参考了哪些产品

---

## 关键原则

- 永远不要一次问超过 **4 个问题**（Q1+Q2+Q3+Q4 可以合并为 1 次回复）
- Q1/Q2 用选择题引导（小白最怕"你随便说"），Q3 用开放描述（让用户说"像什么"），**Q4 强制选择题选技术栈**
- 每个问题附带具体例子，帮用户理解问题在问什么
- 用户的自然语言由 AI 在内部映射到 8 个风格类别，不需要用户知道这些类别的存在
- 如果用户回答模糊，不要追问——用你的判断力给出推荐方案并确认

## 第 2 轮：确认方案（展示你的理解）

根据第 1 轮回答，输出一个简洁的方案确认：

```
我理解你要做的是一个 [产品类型]，主要功能是 [核心功能1-3个]，
面向 [目标用户]，风格偏 [风格方向]。

按照这类产品的标准架构，我建议包含以下页面：
- [页面1]：[一句话说明]
- [页面2]：[一句话说明]
- [页面3]：[一句话说明]
...

这个方向对吗？需要加减什么？
```

**加载项目类型参考：**
→ 读取 `references/project-types/index.md` 匹配项目大类
→ 读取对应的类别文件（如 `references/project-types/01-commercial.md`）获取标准页面清单
→ 如果是 AI 产品，额外读取 `references/ai-products/index.md`

## 提问协议补充规则

### 什么时候该问，什么时候该自己判断

| 场景 | 做法 |
|------|------|
| 用户说"帮我做个网站" | 必须问（信息严重不足） |
| 用户说"帮我做一个类似小红书的App" | 只需确认细节（已有明确方向） |
| 用户说"帮我做一个 SaaS 后台的用户管理页面" | 直接做（信息足够） |
| 用户说"这个页面不好看" | 问具体哪里不满意 + 给 3 个改进方向 |
| 用户发来截图/URL | 分析截图提取设计特征，确认后直接做 |

### 小白用户的特殊处理

如果判断用户是零基础（不懂技术术语、不知道怎么描述需求）：
1. 不用技术术语——说"页面"不说"路由"，说"按钮"不说"CTA"
2. 用具体产品举例——"类似淘宝那样的商品列表"
3. 主动给推荐——"这类产品一般会有XX功能，你需要吗？"
4. 可视化确认——用 ASCII 草图或简单描述帮用户理解布局
5. 分步交付——先做一个页面让他看到效果，再扩展

### 进阶用户的特殊处理

如果判断用户有产品/设计/开发经验：
1. 跳过基础提问，直接进入技术方案
2. 提供更多选择和技术细节
3. 讨论架构决策和 trade-off
4. 直接给代码而不是描述
